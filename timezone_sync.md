# Синхронізація Часових Поясів у SirenUA

Цей документ детально описує архітектуру та логіку синхронізації часу між локальним сервером (SQLite), Firebase (Firestore) та клієнтським iOS додатком (Swift).

---

## 1. Загальний дизайн та Джерело Істини
Для забезпечення консистентності даних уся система SirenUA використовує наступний підхід:
1. **Зберігання на бекенді (SQLite / Firestore)**: Всі мітки часу (`timestamp`) записуються виключно в **UTC** (Універсальний координований час, GMT+0). Це запобігає будь-яким проблемам зі зміною локальних годинників на серверах Render чи серверах Google Firebase.
2. **Локалізація на клієнті (iOS)**: Конвертація з UTC в локальний часовий пояс користувача (зазвичай `Europe/Kiev` — UTC+2 взимку / UTC+3 влітку) відбувається під час рендерингу в інтерфейсі (UI).

---

## 2. Локальна база даних сервера (SQLite)
Усі таблиці аналітики та помилок в `threat_analytics.db` створюються з автоматичним заповненням по UTC:
- Схема таблиць використовує: `timestamp DATETIME DEFAULT CURRENT_TIMESTAMP` (SQLite генерує час UTC).
- Будь-які ручні вставки в Python використовують `datetime.now(timezone.utc).strftime("%Y-%m-%d %H:%M:%S")`.

### Агрегація для Адмін-панелі (Часовий зсув)
Оскільки групування по днях (`daily_stats`) або по годинах (`hourly`) безпосередньо в SQLite-запитах по UTC давало б некоректні дані для України (наприклад, події перших трьох годин доби записувалися б у попередній день), сервер динамічно коригує зміщення:
1. Сервер визначає поточний зсув часового поясу `Europe/Kiev` через `zoneinfo` (наприклад, `+03` влітку або `+02` взимку).
2. Запит SQLite використовує конструкцію:
   ```sql
   SELECT date(datetime(th.timestamp, '+3 hours')) as day, ...
   ```
   *Зсув передається як параметр, розрахований динамічно з Python з безпечним фолбеком на `+03` при помилках імпорту чи відсутності бази часових поясів.*

## 3. Firebase (Cloud Firestore)
Аналітика загрози також дублюється у Firestore для віджетів та додаткових систем.
- Дані пишуться в колекцію `sirenua_history` з полем `timestamp` у форматі UTC стрічки `YYYY-MM-DD HH:MM:SS`.
- **Фільтрація за датою (`/api/history/{region}`)**:
  1. Користувач надсилає локальну дату (наприклад, `2026-07-10` за Київським часом).
  2. Сервер розраховує початок та кінець цієї доби в Києві: `2026-07-10 00:00:00` та `2026-07-11 00:00:00`.
  3. Ці об'єкти конвертуються в UTC: `2026-07-09 21:00:00` та `2026-07-10 21:00:00`.
  4. Запит до Firestore фільтрує події в межах отриманих UTC-значень стрічок, гарантуючи точне відображення доби для України.

## 4. Клієнтська частина (iOS / Swift)
Клієнтський додаток SirenUA отримує стрічки часу UTC від API та форматує їх на пристрої:

### Хронологія користувача (`RegionHistoryEvent.swift`)
Конвертація UTC в локальний час за допомогою Swift `DateFormatter`:
```swift
var displayTime: String {
    let isoFormatter = DateFormatter()
    isoFormatter.dateFormat = "yyyy-MM-dd HH:mm:ss"
    isoFormatter.timeZone = TimeZone(secondsFromGMT: 0) // Вказуємо, що вхідний час є UTC
    if let date = isoFormatter.date(from: timestamp) {
        let timeFormatter = DateFormatter()
        timeFormatter.timeZone = TimeZone.current // Конвертуємо у поточний часовий пояс телефону
        timeFormatter.dateFormat = "HH:mm"
        return timeFormatter.string(from: date)
    }
    return ""
}
```

### Нативна адмін-панель (`AdminDashboardView.swift`)
Функція `formatShortTime` парсить вхідний рядок UTC і конвертує його в локальний часовий пояс користувача, використовуючи системний `TimeZone.current`. Якщо користувач знаходиться в Україні, відображатиметься правильний Київський час.

## 5. Додаткові модулі та евристика
- **Контекст для Gemini**: Модуль `gemini_analyzer.py` перед відправкою повідомлень на ШІ-аналіз додає у системний промпт поточний локальний час у Києві: `ПОТОЧНИЙ КИЇВСЬКИЙ ЧАС: YYYY-MM-DD HH:MM:SS`. Це дає Gemini можливість аналізувати часові вирази (наприклад, "півгодини тому") відносно реального часу атаки.
- **Модифікатор часу доби**: Heuristic-модуль `telegram_monitor.py` визначає поточну годину в Києві, щоб підвищити або знизити оцінку достовірності загрози (наприклад, нічні атаки Шахедів мають вищу ймовірність підтвердження, ніж денні).
